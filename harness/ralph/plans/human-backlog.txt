## 2026-04-26 — #6 RouterTier Pioneer.ai GLiNER2 fine-tune (HUMAN-ONLY)

The runtime code is shipped behind a Protocol with `StubEntityRouter`
fallback. The fine-tune step itself MUST be done by a human in the
Pioneer.ai dashboard — it cannot be done from CLI.

**Files prepared for you (input to Pioneer):**
- `backend/retrieval/router_train/prompt.md` — paste verbatim into
  Pioneer's "Agent prompt" / "System instructions" field.
- `backend/retrieval/router_train/seed_examples.jsonl` — 50 hand
  curated `{query, intent, entities}` examples; upload as Pioneer's
  seed/example data.
- `backend/retrieval/router_train/eval_set.jsonl` — 45 held-out
  items; upload as Pioneer's eval split.
- `backend/retrieval/router_train/README.md` — full step-by-step
  workflow (12 steps, includes synthetic-data generation flow,
  acceptance targets, calibration notes, side-challenge submission
  checklist).

**What you do, in order:**
1. Sign up / log into Pioneer.ai (hackathon onboarding link in the
   README).
2. New Job -> Model = GLiNER2 -> Task = "intent classification +
   NER (multi-task)".
3. Paste body of `prompt.md` into the agent prompt field.
4. Upload `seed_examples.jsonl` as seed data; ask Pioneer to
   generate 500-1000 synthetic examples balanced across the 4
   intents.
5. Run the fine-tune (15-45 min on Pioneer infra).
6. Upload `eval_set.jsonl` as eval split. Targets: intent accuracy
   >= 0.90, macro NER F1 >= 0.85, both beating base GLiNER2 by 10+
   points.
7. Download the fine-tuned weights to
   `backend/retrieval/router_train/weights/<model-name>/` (gitignored).

**After the fine-tune is done — wire it in:**
8. `uv add gliner` (intentionally NOT in pyproject.toml; ~700 MB
   transitive deps via PyTorch + transformers).
9. Set env vars in your shell / `.env`:
   - `QONTEXT_ROUTER=gliner2`
   - `GLINER2_MODEL_PATH="<absolute path to weights dir>"`
10. Verify:
    ```
    QONTEXT_ROUTER=gliner2 GLINER2_MODEL_PATH=... \
      uv run pytest backend/retrieval/tests/test_router.py::TestGLiNER2EntityRouterIntegration -q
    ```
11. Re-run the eval harness with the fine-tuned router live and
    compare against the previous report under
    `backend/eval/reports/`. Fill in the eval-results table at the
    bottom of `router_train/README.md`.

**Pioneer side-challenge submission (€700, best fine-tuned model):**
The README's last section lists the 4-item submission package
(Pioneer job link, README, eval delta, integration code).

**Stub fallback meanwhile:** the cascade runs cleanly without any
of the above — `StubEntityRouter` (regex-based) keeps CI green and
production booting. Real NER recall on natural-language queries is
the upgrade you unlock by completing the steps above.

## 2026-04-26 — #5 AgenticTier Gemini backend (HUMAN-ONLY)

The runtime code is shipped behind the `LLMClient` Protocol with a
`NoopLLMClient` default, so the cascade boots cleanly without a
GEMINI key. To enable real analytical reasoning:

1. Get a Gemini API key from https://aistudio.google.com/apikey.
2. Add to `.env` (or your shell environment):
   ```
   GEMINI_API_KEY=<your key>
   QONTEXT_AGENTIC=gemini
   ```
   Optional: if the same key hits 429 quota due to ingest sharing,
   create a second project key and set
   `GEMINI_API_KEY_AGENT=<key>` then thread it through the
   `GeminiLLMClient(api_key=...)` constructor (current default reads
   `GEMINI_API_KEY`; the secondary-key wiring is a one-line follow-up).
3. Verify (network-required): `GEMINI_API_KEY=... uv run pytest -q
   backend/retrieval/tests/test_agentic.py::TestAgenticTierGeminiIntegration`.
4. Smoke the live endpoint:
   ```
   curl -X POST localhost:8000/api/query \
     -d '{"query":"which customers had support cases handled by emp_1002"}'
   ```
   Expect `tier_used="agentic"` and `relevance >= 0.7` (grounded with
   citations) when both Neo4j is populated and the key is live.

**Defaults meanwhile:** without `QONTEXT_AGENTIC=gemini`, the cascade
uses `NoopLLMClient` — AgenticTier returns a marker text with
relevance 0.3 (ungrounded) so the cascade escalates past it to
`stub`. `analytical` intent queries from the router will still hit
AgenticTier (route_to honored) but produce no real reasoning.

## 2026-04-26 — #8 answer-customer-email follow-ups

The workflow is shipped behind the same `LLMClient` Protocol that
AgenticTier uses, so it boots without a GEMINI key (tests inject
`StubLLMClient`). To wire it up end-to-end:

1. **API endpoint construction.** `POST /api/workflow/answer-customer-email`
   currently only threads `tiers_by_name` through `build_workflow`.
   This workflow needs `llm` + `store` extras — the API route in
   `backend/api/app.py` must be updated to pass them, e.g.:
   ```
   wf = build_workflow(name, engine.tiers_by_name,
                       llm=app.state.llm_client, store=app.state.store)
   ```
   `app.state.llm_client` needs lifespan-time wiring (default
   `NoopLLMClient`, override via `QONTEXT_AGENTIC=gemini`). Without
   this, the workflow endpoint will 400 with a TypeError on missing
   kwargs. (Other workflows that take only `tiers` continue to work
   via the existing call shape.)
2. **Identity-resolution `SAME_AS` clusters.** The workflow currently
   only does deterministic email normalization (lowercase + strip).
   Cluster-aware sender lookup (so `buyer@acme.com` resolves to a
   `Person` node fronted by a `SAME_AS` edge from a `Customer` alias
   node) is the issue's "implementation note" path; punted to a
   follow-up after #11 (identity-resolver upgrade).
3. **Latency benchmarking.** Issue acceptance: p95 ≤ 2s. Needs
   real Neo4j + populated embeddings + a live Gemini key. Add to the
   eval harness as a separate `workflow:answer-customer-email`
   benchmark scenario.
4. **URL slug.** Issue body mentions `/api/workflow/customer_email`
   (underscore); workflow registers as `answer-customer-email`
   (hyphen, matching the issue title and the agreed naming
   convention). Frontend / docs that hardcode the URL must use the
   hyphenated form.
